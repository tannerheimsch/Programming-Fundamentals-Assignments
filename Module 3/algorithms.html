Unit 3: Program Design and Debugging

Algorithms

Here is a definition of the term:

al•go•rithm (wikipedia ref.)

noun

a process or set of rules to be followed in calculations or other problem-solving operations, esp. by a computer : a basic algorithm for division.

How does it work? Start by Defining the problem

One way to think of computer programming is breaking down a process into small, manageable, and programmable pieces. Then we make pieces of a program out of each part of the process. Putting all the pieces together into a working computer program is the next step.

Developing algorithms is a part of the process we go through. This learning activity is about developing those algorithms.

An algorithm is a way of doing something. A shopping list of things we need to do. Assembly instructions for a new bike. Steps in dividing two numbers. Starting a car.

Algorithms in computer programming are more strict than the instructions for assembling a bike. A computer algorithm needs to be followed perfectly, every time. We must provide the correct instructions in terms of sequence, selection, and repetition.

Computers can only do what we tell them to do. So, if we correctly tell them to do something - they do it quickly and correctly every time! And if we screw that up? You can guess what happens.

The First Breakdown

The first thing we do when breaking down a process is to identify three things:

    1. Input: a list of the source data provided to the problem.
    2. Processing: a list of actions needed to produce the required outputs.
    3. Output: a list of the outputs required.

We often start with a problem statement that defines what a program needs to do. Here's an example:

A program is required to read three numbers, add them together and print the total formatted like this:

The total of 1 + 2 + 3 is 6.

That problem statement might be something that's given to us if we're new to programming. But people often expect that we, the software developer, will come up with the problem statements ourselves. And that's where things can get messy; we have to be mindful of all details. And when we don't know something, we need to ask better questions.

But once we have the problem statement, our first step is to identify the nouns in the problem statement. Why?? Because nouns are candidates for inputs and outputs.

A program is required to read three numbers, add them together and print the total.

We can see from this that the inputs are three numbers and the output is the total.

Next, we identify the verbs and adverbs in the statement.

A program is required to read three numbers, add them together and print their total in the following format:

The total of 1 + 2 + 3 is 6.

The verbs help us determine the processing that the programming need to do.

Seems pretty simple, right? Let's do a quick lab:

Lab 1: Algorithms Nouns and Verbs

Defining Diagram

Once you identify nouns you have a very good idea what the inputs and output are. The verbs tell us about processes that must occur. To document inputs, processes, and outputs we use a Defining Diagram.

There are several ways that programmers draw Defining Diagrams. Sometimes, those diagrams are called "IPO Charts" (pronounced "ippo" charts, which rhymes with "hippo charts"). We're not going to do that here. For simplicity we are going to write our defining diagrams in a vertical plain text format.

Here is an example of the style of Defining Diagram that we will use in this class to document and visualize an algorithm. We'll first fill in the inputs and outputs and then add the processing in between.

INPUTS:
  number1
  number2
  number3

PROCESSING:
  read three numbers
  add numbers together
  print total number formatted as "The total of 1 + 2 + 3 is 6."

OUTPUTS:
  total formatted as "The total of 1 + 2 + 3 is 6."

Remember that processing steps are things the program must do. For example, the inputs are pieces of data that will come into the program...but, in processing we must tell the computer to "read" those numbers.

It's important to have all processing steps in the Defining Diagram. So, after you list inputs and outputs, make sure to include any processing that must occur to:

  • get the input (read three numbers)
  • to manipulate or calculate the input to turn it into output (add numbers together)
  • to display the output (print total number)

You do not need to get into details at this point. This is not pseudocode. You are merely getting all of the processes grouped together in the right order. There's no need to get into the math at this point, and there's no need to make it look like code. It's just a list.

Lab 2: Defining Diagrams

Designing a solution algorithm

Next, we design a solution algorithm - which is where we use pseudocode! If this step is done correctly, then writing the actual JavaScript code will go very smoothly.

We use pseudocode here because we don't have to be distracted by the syntax rules of the language. We can just write our thoughts without worrying about the tricky JavaScript (or ANY language) code we may have to write.

Here's how it works. It's really pretty easy!

Start with the defining diagram you did for the problem.

Give the algorithm a name that briefly describes what the algorithm does. It's good to start with a verb. These names can be capitalized and should be camel cased. Here's the name of our algorithm for this exercise:

AddThreeNumbers

That was easy! What's next?

Add an END statement to balance the name of the algorithm (really, just to designate a stopping point).

AddThreeNumbers
    ...
END

That was easy, too! That's the beginning and the end...now for the MIDDLE part...

Pseudocode has a little structure to mimic how it will look in real code. When we add the steps of the algorithm we indent them the same way we indent in JavaScript (For this class, that's two spaces).

Each of the steps relates directly to statements in the process section of the defining diagram. They can be exactly the same, or you can expand the details to make the programming part easier when you get to the JavaScript coding!

And...be sure to put the processing steps in the correct sequence at this point. Thanks to the defining diagram, much of that sequencing has already been done.

So, here's what our pseudocode might look like:

AddThreeNumbers
  Prompt for number1
  Prompt for number2
  Prompt for number3
  Set total = number1 + number2 + number3
  Display total
END

Of course, you should get more verbose than that. Here's much better pseudocode:

AddThreeNumbers
  Initialize number1 to 0
  Initialize number2 to 0
  Initialize number3 to 0
  Initialize total to 0

  Prompt for number1 with "Enter the first number"
  Prompt for number2 with "Enter the second number"
  Prompt for number3 with "Enter the third number"

  Set total = number1 + number2 + number3

  Display "The total of " + number1 + " + " +
          number2 + " + " + number3" +
          " is " + total + "."
END

Notice that this version includes the text the user will see. It also includes the algorithm used to construct the output that gets displayd on the screen. Pseudocode is a planning tool more than anything, and if you find that this helps you "visualize" the program, then by all means be more specific. This second, more expressive form is preferred for assignments.

Lab 3: Creating Solution Algorithms

Checking the solution algorithm

One of the major advances in computer programming has come just in the past few years. It's called Test Driven Development or TDD. This is very important for new programmers because most of the local employers are implementing TDD at some level in the programming shops. (wikipedia ref.)

With TDD, you first write code that will test your real code. Then you write the real code. When you run the testing code it will use your real code just like in the final application. If it runs correctly then it has passed the test. One benefit of this process is that you get a bunch of tests you can use to prove your code works. But TDD is more than just some checks - it's part of the design process. By writing test cases first, you're forced to think more about how your code will work and how your algorithm is designed.

We're going to start thinking about TDD by using the "Desk Checking" concept. This is a manual process that you do "on your desk". It won't be automatic, but you're not expected to do automatic TDD yet. We're just getting you ready to do TDD without doing the coding part. Eventually, you'll get the chance to play with real test cases.

Steps in Checking an Algorithm

  1. Choose 2 or 3 simple input test cases that are valid.
  2. Establish what the expected result should be for each test case.
  3. Write the relevant variables in the testing section of the project or lab. We'll have a section of our files where we write down the variables, the test data, the expected results, and the actual results when we're done with the code.
  4. Then test the algorithm by running your test cases through it. You just do the calculation without any computer code, use a calculator if you need to. Then write down your expected results.
  5. Repeat this for all your test data.

Example:

Here's the pseudocode for the example we've been looking at:

ALGORITHM:

AddThreeNumbers
  Initialize number1 to 0
  Initialize number2 to 0
  Initialize number3 to 0
  Initialize total to 0

  Prompt for number1 with "Enter the first number"
  Prompt for number2 with "Enter the second number"
  Prompt for number3 with "Enter the third number"

  Set total = number1 + number2 + number3

  Display "The total of " + number1 + " + " +
          number2 + " + " + number3" +
          " is " + total + "."
END

And here are our test plans:

VARIABLES: number1, number2, number3, total

TEST CASE 1:

  INPUT VALUES: 10, 20, 30

  EXPECTED RESULT: "The total of 10 + 20 + 30 is 60."

  ACTUAL RESULT: Fill this in.

TEST CASE 2:

  INPUT VALUES: 40, 41, 42

  EXPECTED RESULT: "The total of 40 + 41 + 42 is 123."

  ACTUAL RESULT: Fill this in.

Note that we list the input variables, in the order of our inputs. This makes it easy to see which number corresponds to which input variable.

Once we turn the solution algorithm into working code, we would come back and fill in the actual data. However, we might discover that we have an issue with our code: JavaScript requires that we convert the data we got from the user, which comes to us as strings, into numbers before we can add the numbers together. So we may modify our pseudocode so it looks like this: 

AddThreeNumbers
  Initialize number1 to 0
  Initialize number2 to 0
  Initialize number3 to 0
  Initialize total to 0

  Prompt for number1 with "Enter the first number"
  Prompt for number2 with "Enter the second number"
  Prompt for number3 with "Enter the third number"

  Convert number1 to a number
  Convert number2 to a number
  Convert number3 to a number

  Set total = number1 + number2 + number3

  Display "The total of " + number1 + " + " +
          number2 + " + " + number3" +
          " is " + total + "."
END

There. Much better pseudocode. Now we have a well-designed solution algorithm we can transform into JavaScript code.

Going forward, you'll need to provide at least three test plans as part of your program design. These plans should test various aspects of your program so they are an accurate test of the way the program should respond. Don't let your users be your testers!

This would be a good time for you to crack open your text editor and retype this pseudocode. Then convert it into JavaScript code for practice.
