JavaScript Coding Standards

Here are the coding standards for the course. Your projects and labs will be graded partially on conformance to these rules. This document will be expanded as the semester progresses.

One of the most important aspects of modern programming is readability. To be able to read code written by other people and understand it has become much more important than having condensed, optimal code. Modern programming languages can optimize your code better than you can. So, the number one criteria for the way we write is for other human beings to read it.

Indentation

The unit of indentation of JavaScript source code will be two (2) spaces. Do not use the tab character for indenting. The course editor (Notepad++) should be configured so that when you press the tab key it will insert spaces instead of a tab character. Keep this setting and if you use another text editor make sure you are not inserting tab characters into your code.

All of the code examples in these learning units are formatted with 2 spaces. Code in videos or code from the web may have different indentation. Text editors are often set up to use tabs instead of spaces. Code that you write should use spaces, not tabs, and it should be consistent.

Here's an example of poorly formatted JavaScript code using no indentation.:


var age = 16;

if(age >= 16){
document.write("You can drive a car!");
}else{
document.write("You are not old enough to drive.");
}

Here's an example of poorly indented JavaScript code using both two and four spaces. Don't do this!:


var age = 16;
if(age >= 16){
    document.write("You can drive a car!");
}else{
  document.write("You are not old enough to drive.");
}

Notice how things don't line up at all?? Be consistent with your indentation and ensure things line up.

Here's an example of properly indented JavaScript code with two spaces:


var age = 16;
if (age >= 16) {
  document.write("You can drive a car!");
} else {
  document.write("You are not old enough to drive.");
}

At the end of the course, indentation will be very important in terms of code readability because your code will become much more complex. You'll have multiple levels of indentation, much like an HTML page's code. You'll be reminded of indentation throughout the course.

Line Length

Avoid making any line of JavaScript code more than 80 characters. This improves readability. Sometimes this can't be avoided but when possible it is required to have 80 or fewer characters in a line of code.

On today's monitors, which are very wide, you might be wondering why we'd use this artificial constraint. The reason is that other developers might want to place the code editor next to the browser to see both at once, and horizontal scrolling is annoying.

Variable Declarations

Good variable naming and usage is very important when learning to program computers. Here is a list of the rules for variables:

  1. Variables must follow the naming conventions outlined below.
  2. All variables must be declared before they are used. Many programming languages require this as part of their syntax. However, JavaScript does not require this so you need do this intentionally.
  3. All variables must be declared with the var keyword.
  4. var statements should be at the top of your JavaScript code, or at the top of the function if the variable is used only inside of the function.
  5. Every variable should be declared on its own line. While JavaScript allows you to declare more than one variable on a single line, our convention will be one variable declaration per line.
  6. Variables should be listed in alphabetical order.
  7. Every variable should have a descriptive name that defines its purpose. See section on Variable names.

Naming Conventions

How things are named in a computer program is very important. In the early days of programming names were very short and cryptic to save on RAM usage. You may have seen variables with names like "i", "fnme", and "cal_prc". These days our computers all have lots and lots of RAM and we can stop this old confusing habit. Here are the course rules for naming things:

  1. Names should be created with the 26 upper and lower case letters (A..Z, a..z) and the 10 digits (0..9).
  2. Do not use spaces, the dollar sign "$", or slashes "/ \" in names. Underscores are acceptable, but not widely used in JavaScript.
  3. Variables and functions should start with a lower case letter.
  4. Constructor plain functions should start with an upper case letter.
  5. Abbreviations should be avoided, use complete words wherever possible, which is the vast majority of the time. What does "cal_prc" mean? Calculate Price? Calculate Percentage? Calender Perfection? Names need to be very clear, for others and for you. You won't remember your own cryptic names in the future.
  6. Names that have more than one word should use the camelCase ("camel case") convention. The name starts with a lower case letter and then each new word in the name starts with an upper case letter. Examples:

Comments

Good documentation will be required in your labs and projects files. Here is a minimal list of what the instructor will be looking for. Quality counts!

  1. The student's name at the top of lab and project files.
  2. All functions need to have a block of comments describing what the function does. More details later when we get to functions.
  3. All loop constructions will have a comment block immediately preceding the loop. The comments should describe what the loop accomplishes and why it exists.
  4. Use descriptive variable names for everything and keep comments to a minimum, expressing why things are there and what your thoughts are, not what the code does.

Here's an example of bad comments:


// This loops 5 times and prints "Hello World" each time.
for (var x = 0; x < 5; x++) {
  document.write("Hello World\n");
}

First, we know it's a loop. The use of for tells us that. Second, we don't know what x is. This could be better expressed this way:


// we need to display "Hello World" five times per the program's requirements.
for (var counter = 0; counter < 5; counter++) {
  document.write("Hello World\n");
}

This comment expresses why we're doing something. It also has much more clear variable names.

Semicolons

Semicolons are used in JavaScript to end a line. Most of the time they are not required by JavaScript. However, they are required in a few places so as a course standard, that is also a convention in the industry, the use of semicolons at the end of statements is required.

Examples:

// use of Semicolons

var name;
name = "Fred";
return name;

var total = 0;
total = total + 100;

There are exceptions to the rule!

There are some statements that should not have semicolons. These are usually compound statements like this:


// semicolons in "if" statements

if (something === 1) {   //<-- No semicolons here!
  // Code in here will have semicolons.
} else {   //<-- No semicolons here!
  // Code in here will have semicolons too!
} //<-- No semicolons here!

A semicolon would cause an error if they are added here where it says they shouldn't:


// Incorrect semicolons

if (something === 1) {
  x = 5;
  name = "Fred";
} else {;   // <-- A semicolon here will cause an error
  x = 10;
  name = "Bill";
}

Statements

The way we write statements is critical for code that is readable and easy to understand. Here is how we're going to write JavaScript statements.

Simple Statements

  • Every simple statement must end with a semicolon.
  • Every simple statement must be on its own line. This code will work but it fails the course standards in a couple of ways:

// WRONG WAY!

var name = "Fred"; var city = "Madison";

document.write(name + ", " + city)
  • On line 3 there are two statements. JavaScript doesn't care but your instructor and future professional teammates will.
  • On line 5 the statement is missing the semicolon at the end. JavaScript won't remind you to put them there. You will just have to get in the habit.

Here's the correct version:


// Right way!
var city = "Eau Claire";
var name = "Homer";

document.write(name + ", " + city);

Compound Statements

Compound statements are different than simple statements. They are statements that contain lists of other statements enclosed in curly braces "{}".

  • Enclosed statements should be indented.
  • The left { should be at the end of the line that starts the compound statement and should have a space in front of it.
  • The right } should begin a line and should line up with the beginning of the compound statement.
  • Braces ({}) should be used on all enclosed statements, even if there is only one enclosed statement.

Here are some examples:


// "if" statements

if (condition) {
  // statements
}

if (condition) {
  // statements
} else {
  // statements
}

if (condition) {
  // statements
} else if (condition) {
  // statements
} else {
  //statements
}

// for loops
for (count = 0; count < 5; count ++) {
  // statements
}

// while loops
while (count < 5) {
  // statements
  count++;
}


// function
function printName(theName) {
  document.write(theName);
}

Whitespace

Whitespace can improve the readability of code. Here are the course standards for whitespace.

  • Use blank lines liberally, your code shouldn't be all bunched together.
  • A keyword followed by a left parenthesis "(" should be separated by a space.
  • Left braces should have a space in front as well.

// Spaces in logic statements

if (condition) {

while (true) {
  • Most operators should have a space on either side.

// Operator spacing

var name = "Fred";
var x = 1;

x = x + 1;

if (name === "Fred") {
  • A space should follow every comma ","

Those are the coding standards for the course. You are expected to know them and be able to demonstrate them on every assignment.
